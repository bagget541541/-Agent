# Mode B 输入文档处理方案

> 创建日期：2026-05-31  
> 状态：待实施

---

## 一、背景

Mode B（合并已有文档）目前接两种来源的 Word 文档作为输入：

| 文档类型 | 来源技能 | 产生时机 |
|---------|---------|---------|
| **公告分析.docx** | `news-analyzer`（Step 2） | 银行公告抓取 + LLM 分析后输出 |
| **微信文章解析.docx** | `wechat-article-extractor`（Step 1） | 微信公众号文章提取后输出 |

两种文档的**结构差异极大**，当前 Mode B 的 `read_docx_content()` 只能识别 Heading 1 → Heading 2 层级，导致公告分析和微信文章的结构化信息丢失。

---

## 二、两种文档结构对比

### 2.1 公告分析文档（农行0529公告分析.docx）

```
文档标题："公告分析报告"
├── [Heading 2] 1. 关于发布部分信用卡...（公告条目1）
│   ├── 原文链接：https://...
│   ├── [Heading 3] 消息内容
│   │   └── （纯文本段落）
│   └── [空行]
├── [Heading 2] 2. 关于调整信用卡...（公告条目2）
│   ├── 原文链接：https://...
│   ├── [Heading 3] 消息内容
│   │   └── （纯文本段落）
│   └── ...
└── ...
```

**特征**：
- 无 Heading 1，直接 Heading 2
- 内容以纯文本段落为主
- 图片极少或无
- 字段以 Heading 3（如"消息内容"）和普通段落（原文链接）呈现

### 2.2 微信文章解析文档（微信文章解析_南京秦农_0531.docx）

```
├── [Heading 1] 南京秦农银行信用卡...（文章标题）
│   ├── **银行：** 南京秦农银行
│   ├── **标题：** 秦农信用卡权益升级...
│   ├── **发布时间：** 2026-05-25
│   ├── **关键信息：** （结构化文本）
│   ├── **点评：** （分析文本）
│   ├── [图片] 文章内嵌图
│   └── **原文链接：** https://...
└── ...
```

**特征**：
- 有 Heading 1（文章标题字段）
- 内容以 **字段名加粗** 方式呈现（`银行：` `标题：` `关键信息：` 等）
- 包含内嵌图片
- 字段名在 `KNOWN_FIELD_NAMES` 中有定义

---

## 三、当前问题

| 问题 | 影响 |
|------|------|
| `read_docx_content()` 要求 Heading 1 → Heading 2 层级 | 公告分析文档只有 H2 无 H1 → 条目无法自动归入分类 |
| 字段加粗（`银行：` `关键信息：`）不被识别为结构化信息 | 微信文档的结构化字段丢失 |
| 两种文档同时输入时，无法区分类别 | 合并结果混乱 |
| 图片位置保留不完整 | 微信文章图片在合并后可能错位 |

---

## 四、改造方案

### Phase 1 — 自适应解析器（核心）

增强 `read_docx_content()`，实现三级自适应：

```
                        ┌─ Heading 1/2 层级清晰 → 原逻辑不变
     输入文档 ──────────┼─ 只有 Heading 2 无 H1  → "公告模式"
                        └─ 无 Heading，字段加粗   → "字段模式"
```

#### 公告模式（自动识别）

| 规则 | 行为 |
|------|------|
| 检测到 Heading 2 但无 Heading 1 | 自动创建 `"银行公告"` 作为 H1 容器 |
| 每条 H2 → 一个公告条目 | 原文链接、消息内容等归属到当前 H2 |
| 若文本中包含 `银行：` | 用银行名替换 H1 标题（如 `"农行公告"`） |

#### 字段模式（自动识别）

| 规则 | 行为 |
|------|------|
| 扫描所有段落，检测是否含 `KNOWN_FIELD_NAMES` + 冒号 | 匹配率 > 20% → 进入字段模式 |
| `银行：xxx` | 创建 H1 容器，标题 = 银行名 |
| `标题：xxx` | 创建 H2 条目 |
| `关键信息：` / `点评：` / `消息内容：` | 归入当前 H2 的 content 列表 |
| `原文链接：` | 存入 H2 的 `url` 字段 |
| `发布时间：` / `作者：` | 存入 H2 的元数据字段 |
| 其他段落 | 归入当前 H2 的 content |
| 图片 | 按顺序关联到最近的 H2 |

#### 自动模式切换阈值

```
if h1_count == 0 and h2_count > 0:
    公告模式
elif bold_field_ratio > 0.2:   # 字段加粗行占比 > 20%
    字段模式
else:
    标准模式（当前逻辑）
```

### Phase 2 — 跳过后半段持卡分析（Mode B + C 交叉）

`merge_docs.py` 增加 `--skip-analysis` 参数：

| 参数 | 行为 |
|------|------|
| 无 `--skip-analysis` | 完整流程：读 → 合 → 持卡分析 → 输出 docx |
| `--skip-analysis` | 读 → 合 → 输出 JSON + MD |

输入文档经过 Merge 后，输出**带结构化层级和图片位置的 JSON**，可直接给 `generate_report.py` 消费。

### Phase 3 — 图片保留增强

| 措施 | 实现 |
|------|------|
| 原文顺序保留图片位置 | `read_docx_content()` 返回 `image_positions: [{before_para_idx, after_para_idx}]` |
| 去重相同图片 | 按 byte 哈希去重，相同图片只保留一份 |
| 合并时保持图片索引一致 | `merge_contents()` 重建 `all_images` 时重映射 rId |

### Phase 4 — 去重与合并策略

| 场景 | 策略 |
|------|------|
| 两篇文章标题相似度 ≥ 0.85 | 保留内容更完整的一篇，丢弃重复 |
| 同一银行 + 同一条公告 | 合并 key 信息，去除冗余 |
| 微信文章 + 公告文档包含同一主题 | 保留两边，标注"来源：微信"和"来源：官网公告" |

## 五、实施顺序

```
D1 ── Phase 1: read_docx_content() 自适应解析
 │     ├── 公告模式（H2 → 自动 H1）
 │     └── 字段模式（KNOWN_FIELD_NAMES 解析）
 │
D2 ── Phase 2: merge_docs.py --skip-analysis
 │     ├── 参数支持
 │     └── 联动 _agent.py Mode C
 │
D3 ── Phase 3: 图片保留与排序
 │     ├── image_positions 追踪
 │     └── merge 时重映射
 │
D4 ── Phase 4: 去重与智能合并
 │     ├── title 相似度检测
 │     └── 银行 + 标题联合去重
 │
D5 ── 测试 & 文档
       ├── 用实际文档端到端验证
       └── 更新 README / CHANGELOG
```
