# 多主题公众号文章拆分设计

## 1. 目标

本设计用于解决当前公众号文章抽取链路默认“一篇文章对应一条资讯”的局限，使系统能够同时适用于：

- **单主题文章**：一篇只讲一个活动 / 一个新卡 / 一个权益变更 / 一个公告
- **多主题文章**：一篇中包含多个活动、多个卡种、多个权益变更，或混合类型主题合集

设计目标不是单纯“拆更多条”，而是建立一个**可闭环、可回退、可渐进迭代**的处理链路，使其满足以下要求：

1. 单主题文章不会因引入拆分机制而变差
2. 多主题文章能自动拆成多个独立主题条目
3. 每个主题条目都能继续走现有标准化、审核、Word 生成链路
4. 能与现有 `content_blocks / normalizer / review_queue / Word` 方案兼容
5. 即使拆分失败，也能安全回退为单条输出，不影响产物可用性

---

## 2. 现状问题

当前链路默认：

```text
一篇公众号文章 -> 一条 CreditCardItem
```

这在单主题文章上成立，但在多主题文章上会出现以下问题：

### 2.1 分类失真
一篇文章中既有活动又有权益变更时，整篇只能落一个分类，导致下游字段模板错位。

### 2.2 结构化字段互相污染
例如：
- 第一部分讲“积分活动”
- 第二部分讲“新卡上市”

若仍强行生成一条 item，则：
- `活动内容` 可能混入新卡介绍
- `详情` 可能混入活动规则
- `亮点摘要` 可能只抓到其中一个主题

### 2.3 Word 最终仍需人工拆分
自动化没有减少真正的人工负担，只是把“拆主题”的工作留到 Word 阶段处理。

### 2.4 OCR / 图片块干扰更严重
多主题文章通常伴随多张图、多段说明，若不先按主题切分，`ocr_fact` 和正文容易跨主题污染。

---

## 3. 核心设计原则

### 原则 1：先切块，再识别主题，再标准化
不要对整篇直接分类和抽取，而应改成：

```text
文章 -> 内容块 -> 主题候选 -> 主题单元 -> 标准化条目
```

### 原则 2：拆分是增强，不是替代
如果文章没有明显多主题特征，系统必须能够自动退回单主题流程。

### 原则 3：主题拆分与分类解耦
“是否多主题”先判断，“每个主题是什么类别”后判断，不要混成一步。

### 原则 4：保留文章来源关系
即使拆成多个主题条目，也要保留它们源自同一篇文章，以便追溯、审核和 Word 展示。

### 原则 5：低置信度时宁可少拆，不要乱拆
错误地把一篇单主题文章拆碎，成本通常高于把一篇多主题文章暂时保守为单条。

---

## 4. 闭环处理链路

推荐把公众号链路改成以下 6 步：

### Step 1：抓取文章原始内容
输出 `ArticleEnvelope`，保存：
- 原始标题
- URL
- 公众号名
- 发布时间
- 原始 HTML / 正文文本
- 图片列表
- OCR / 视觉提取结果

### Step 2：生成 `content_blocks`
把整篇文章拆成内容块，而不是直接拼成一个全文字符串。

### Step 3：判断是单主题还是多主题候选
基于 block、标题、段落结构、重复模板特征进行判定。

### Step 4：若为多主题候选，则切分 `TopicCandidate`
把文章拆成 1..N 个主题候选块。

### Step 5：每个主题候选独立标准化
每个主题候选都走统一 normalizer，输出标准 `CreditCardItem`。

### Step 6：汇总到 batch，进入 review / Word
即：
- 单主题文章 → 1 条 item
- 多主题文章 → 多条 item

这样就形成闭环：

```text
公众号文章
  -> ArticleEnvelope
  -> content_blocks
  -> TopicCandidate[]
  -> CreditCardItem[]
  -> review_queue + Word
```

---

## 5. 数据结构设计

## 5.1 ArticleEnvelope

建议在公众号抓取后引入中间结构：

```python
{
  "article_id": "wx_20260531_001",
  "url": "https://mp.weixin.qq.com/s/...",
  "publisher_name": "农行信用卡",
  "publish_time": "2026-05-31",
  "raw_title": "618福利合集",
  "raw_html": "...",
  "content_blocks": [...],
  "topic_split_meta": {
    "is_multi_topic_candidate": True,
    "split_confidence": 0.81,
    "signals": ["multiple_numbered_headings", "repeated_activity_templates"]
  }
}
```

### 说明
- `article_id`：同篇多主题条目共享
- `content_blocks`：后续主题识别的唯一基础输入
- `topic_split_meta`：记录为何判为多主题候选

---

## 5.2 ContentBlock

建议最小字段如下：

```python
{
  "block_id": "b12",
  "type": "article_text|ocr_fact|ocr_noise|image_cta|image_caption|separator",
  "text": "活动一：618积分五倍赠",
  "position": 12,
  "image_index": 2,
  "is_heading_like": True,
  "topic_hint": "活动",
  "bank_hint": "农业银行",
  "confidence": 0.88
}
```

### 推荐 block type
- `article_text`：正文段落
- `ocr_fact`：图片中识别出的有效事实
- `ocr_noise`：无价值图片说明
- `image_cta`：扫码申卡、关注公众号等 CTA
- `image_caption`：图片标题或图注
- `separator`：分隔线、明显区隔标记

### 说明
`content_blocks` 不只是为了去噪，更是为了支持主题边界识别。

---

## 5.3 TopicCandidate

主题候选结构：

```python
{
  "topic_id": "wx_20260531_001_t2",
  "article_id": "wx_20260531_001",
  "source_article_title": "618福利合集",
  "start_block": 8,
  "end_block": 14,
  "headline": "618积分五倍赠",
  "blocks": [...],
  "topic_type_hint": "活动",
  "split_confidence": 0.84,
  "split_signals": ["heading_marker", "activity_template"]
}
```

### 说明
- `headline` 是主题内标题，不等于整篇标题
- `topic_type_hint` 只是提示，不等于最终分类
- `split_confidence` 用于低置信度回审

---

## 6. 单主题 / 多主题统一处理策略

## 6.1 判定目标
系统不应该假设“所有文章都拆”，而应该先做判定：

- `single_topic`
- `multi_topic_candidate`

### 推荐策略
- **默认保守**：先视为单主题
- 只有当命中足够强的多主题信号时，才升级为多主题候选

---

## 6.2 多主题候选识别信号

建议分为强信号和弱信号。

### 强信号
任一命中即可高度怀疑多主题：
1. 出现多个明确编号小标题
   - `一、二、三`
   - `1. 2. 3.`
   - `01/02/03`
   - `活动一/活动二`
2. 同篇中出现多组重复模板：
   - `活动时间 / 活动对象 / 活动内容`
   - `卡种 / 权益 / 适用人群`
3. 同篇中存在多个明显不同的主题标题
4. 同篇中多个图文段落各自形成独立闭环（标题 + 说明 + 图片）

### 弱信号
多个同时命中才升级：
1. 多个不同活动名
2. 多个不同卡种名
3. 多个动作词切换：
   - `首发`
   - `新增`
   - `调整`
   - `返现`
   - `积分`
4. 段落之间主题关键词切换明显

### 判定输出示例

```python
{
  "is_multi_topic_candidate": True,
  "split_confidence": 0.79,
  "signals": [
    "multiple_numbered_headings",
    "repeated_activity_templates"
  ]
}
```

---

## 6.3 单主题回退机制

若不满足多主题条件，直接生成一个全量 `TopicCandidate`：

```python
[
  {
    "topic_id": "wx_20260531_001_t1",
    "article_id": "wx_20260531_001",
    "headline": raw_title,
    "blocks": full_content_blocks,
    "split_confidence": 1.0,
    "split_signals": ["single_topic_fallback"]
  }
]
```

### 这点很关键
这保证：
- 单主题文章不会因为引入主题拆分而劣化
- 整条链路天然闭环
- 下游不用区分单主题和多主题，统一消费 `TopicCandidate[]`

---

## 7. 主题切分算法设计

## 7.1 总体思路
采用“**先找主题起点，再向后吸附**”的方法，而不是对所有段落做复杂聚类。

```text
blocks -> topic start markers -> merge until next marker -> TopicCandidate[]
```

---

## 7.2 主题起点识别

一个 block 若满足以下任一条件，可视为主题起点候选：

### 标题型起点
- `is_heading_like == True`
- 文本短且集中表达主题
- 含编号前缀，如：
  - `一、618积分五倍赠`
  - `活动二：新客返现`
  - `卡种一：Visa世界杯版`

### 模板型起点
- 紧接着出现成组字段：
  - `活动时间`
  - `活动对象`
  - `活动内容`
- 或 `卡种 / 权益 / 亮点 / 年费` 成组出现

### 图片型起点
- 新图片 OCR 中出现新的卡种名 / 新活动名 / 新银行名
- 且与前一主题 hint 不一致

### 关键词切换型起点
前一块以活动为主，当前块出现：
- `关于...授权书`
- `正式发行`
- `新卡上市`
- `权益调整`
可视为进入新主题

---

## 7.3 主题终点识别

一个主题从起点开始向后吸附，直到满足以下任一条件：

1. 遇到下一个高置信度主题起点
2. 遇到明确的文章结束/总结块
3. 遇到分隔符且后续块有更强的新主题信号

### 吸附规则
建议把以下 block 自动并入当前主题：
- 普通正文段
- 与当前主题 bank_hint 一致的 OCR fact
- 介绍/说明性 image_caption

建议不并入：
- `ocr_noise`
- `image_cta`
- 明显属于下一主题的标题块

---

## 7.4 主题合并修正

初步切完后需要做一次后处理，避免过拆。

### 需要合并的情况
1. 只有一个很短标题块，后面正文很少
2. 两个相邻主题同属一个活动，只是被图片打断
3. OCR fact 被误拆成独立主题，但没有正文支撑

### 合并策略
若相邻两个 `TopicCandidate` 同时满足：
- 类别 hint 一致
- 银行 hint 一致
- 后一个主题长度过短
- 后一个主题没有独立 headline

则自动合并。

---

## 8. 标准化接入方式

## 8.1 不直接从 Article 标准化，而是从 TopicCandidate 标准化

把当前：

```text
article -> normalizer -> item
```

改成：

```text
topic_candidate -> normalizer -> item
```

### 好处
- 单主题/多主题统一接口
- normalizer 不需要知道“原文是不是一篇多主题合集”
- 所有 review / Word 逻辑都无需改模型理解方式

---

## 8.2 TopicCandidate 到 CreditCardItem 的映射

建议 normalizer 接收：

```python
normalize_topic(topic_candidate, article_meta) -> CreditCardItem
```

其中：
- `raw_title` 可优先取 `topic_candidate.headline`
- `raw_text` 来自该 topic 的正文块拼接
- `content_blocks` 只传该 topic 范围内的 blocks
- `issuer/source/category/structured/display` 都只基于该 topic 计算

---

## 8.3 需要保留的追溯字段

每条拆出来的 item 建议增加：
- `article_id`
- `topic_id`
- `source_article_title`
- `topic_split_confidence`
- `topic_split_signals`

这样后续：
- review queue 可追溯到原文章
- Word 如需展示“同篇来源”也能实现
- 低置信度拆分可单独复核

---

## 9. 对现有模块的改造建议

## 9.1 `fetch_wechat_article.py`

### 当前问题
更偏向直接输出全文文本 + 图片结果，适合单主题，不适合多主题拆分。

### 建议改造
1. 输出 `ArticleEnvelope`
2. 先生成 `content_blocks`
3. 图片/视觉结果不要直接拼 `full_text`
4. 为每个 block 打基础标签：
   - `type`
   - `position`
   - `is_heading_like`
   - `topic_hint`

### 产出目标
抓取层只负责“采集”和“块化”，不负责最终主题判断。

---

## 9.2 新增 `common/topic_splitter.py`

建议新增统一主题拆分模块。

### 核心接口

```python
detect_multi_topic(article_envelope) -> dict
split_article_into_topics(article_envelope) -> list[TopicCandidate]
merge_small_topics(topics) -> list[TopicCandidate]
```

### 职责
- 判断单主题/多主题候选
- 执行主题切分
- 执行过拆修正
- 输出拆分信号与置信度

这是整个设计中最关键的新模块。

---

## 9.3 `common/normalizer.py`

### 改造方式
从“接收 article”改成“接收 topic candidate”。

### 需要调整
1. title/display 字段改基于 topic headline 生成
2. structured 只从 topic blocks 提取
3. confidence / review flags 中新增：
   - `needs_topic_split_review`
4. 若 topic_split_confidence 低，则进入 review queue

---

## 9.4 `_agent.py`

### 改造方式
公众号流程改成：

```text
抓文章 -> build blocks -> split topics -> normalize each topic -> add to batch
```

而不是：

```text
抓文章 -> 直接生成一个 item
```

### 好处
- 逻辑更清晰
- 与官网来源的 item 流统一
- 便于后续做 review queue

## 10. 单主题与多主题的闭环保证

这是本设计最重要的一点：**同一套实现必须同时适用于单主题和多主题。**

## 10.1 单主题闭环
流程：

```text
article -> content_blocks -> detect_multi_topic(False) -> 生成单个 TopicCandidate -> normalizer -> item
```

### 结果
- 行为与当前系统接近
- 不会因为新机制引入过多额外风险
- 可以无痛迁移

---

## 10.2 多主题闭环
流程：

```text
article -> content_blocks -> detect_multi_topic(True) -> split topics -> normalize each -> items[]
```

### 结果
- 一篇文章拆出多条 item
- 每条 item 可独立分类、审查、进入 Word
- review queue 也能标记拆分置信度

---

## 10.3 失败回退闭环
即便拆分失败，也必须可回退。

### 回退规则
若出现以下情况：
- 没找到稳定主题边界
- 切分后所有 topic 都过短
- 多主题信号弱、拆分置信度不足

则：
- 回退成单个 `TopicCandidate`
- 打 `needs_topic_split_review`

### 结果
系统不会因为拆分器不成熟而直接崩或产出不可用内容。

---

## 11. 审核与 Word 输出如何配合

## 11.1 审核层新增的 flag
建议新增：
- `needs_topic_split_review`
- `possible_multi_topic_article`
- `topic_boundary_low_confidence`

### 审核用途
当系统对是否多主题、怎么切主题不确定时，人工只看这些条目。

---

## 11.2 Word 层无需理解复杂拆分过程
Word 只需要把最终 item 当普通资讯输出。

### 可选增强
可在必要时保留信息：
- 同篇来源标题
- 同篇多个主题的原始文章链接相同

例如可选展示：
- 来源：农业银行（同篇专题拆分）

但这不是必须项。

---

## 12. 最小可落地实施方案

为确保快速闭环，建议按两阶段做。

## 第一阶段：只支持“明显多主题”文章

### 支持的强信号
1. 多个编号小标题
2. 多组活动模板字段
3. 多个明显主题标题
4. 多个独立图文小节

### 暂不处理
- 极弱语义切换
- 无标题、无编号、纯长文混合主题

### 目标
先解决最常见、最容易人工拆分的多主题文章。

---

## 第二阶段：增强弱信号拆分能力

### 增加能力
1. 基于主题关键词切换拆分
2. 基于 OCR fact 聚类辅助拆分
3. 基于同类字段重复出现做边界修正

### 目标
提升复杂合集文章的拆分率，但不牺牲稳定性。

---

## 13. 实施步骤清单

### P0：打通闭环必需能力
1. 新增 `ArticleEnvelope` 结构
2. 将公众号抓取输出改为 block 化
3. 新增 `common/topic_splitter.py`
4. 实现单主题 fallback
5. 实现 topic -> item 标准化接口

### P1：实现明显多主题拆分
6. 编号标题识别
7. 模板字段组识别
8. 主题起点/终点识别
9. 相邻短 topic 合并
10. review flags 接入

### P2：增强复杂场景
11. OCR fact 辅助主题边界识别
12. 弱信号多主题判定
13. Word 可选展示同篇来源信息
14. 统计拆分命中率和误拆率

---

## 14. 验收标准

按本设计实施后，应满足以下闭环标准：

### 单主题文章
- 能稳定输出 1 条 item
- 输出质量不低于当前方案
- 不因主题拆分机制而明显劣化

### 多主题文章
- 对明显多主题文章能输出 2..N 条 item
- 每条 item 的分类、标题、structured 不再互相污染
- Word 中不再需要人工先拆主题再修内容

### 系统稳定性
- 拆分失败时能回退为单条输出
- review queue 能标出需要人工确认的拆分样本
- 同一篇文章拆分出的多条 item 可追溯到原文章

---

## 15. 结论

最推荐的公众号抽取升级方向不是继续增强“整篇文章抽一条 item”的规则，而是：

**把公众号处理链路升级为：文章容器 -> 内容块 -> 主题候选 -> 主题条目 -> 标准化输出。**

这套方案的关键价值在于：

1. **对单主题兼容**：不破坏当前主流程
2. **对多主题有效**：真正减少人工拆分工作
3. **能闭环落地**：拆分后的主题仍能走现有标准化、审核、Word 输出链路
4. **便于渐进迭代**：可以先支持明显多主题，再逐步增强复杂场景

因此，这是一条适合当前项目现状、且能兼顾短期可落地与中期可扩展的方案。
