# 下一阶段重构蓝图

## 1. 背景与目标

本项目是自用型信用卡资讯周报流水线，核心目标不是“生成一份能看的 Word”，而是：

1. 尽量减少人工逐条审核、逐段修改的时间
2. 让最终 Word 内容更稳定、更像可直接交付的编辑稿
3. 让问题尽可能在上游暴露并被修复，而不是拖到 Word 阶段临时补救
4. 让后续新来源、新银行、新类型内容接入时，维护成本可控

当前从最终 Word 暴露出来的问题主要包括：
- 标题不够简洁，核心主题不突出
- 来源显示不自然，银行识别不稳定
- OCR 噪音混入正文/详情/亮点
- 官网导航页、页脚、热线等污染正文
- 分类不稳定，导致下游展示逻辑错位
- Word 生成脚本承担了过多内容修复职责
- 人工审核仍然需要通读全文，而不是只看异常项

本蓝图的设计原则是：**从 Word 中暴露的问题反推上游架构问题，优先建设“稳定中间层 + 可审核机制”，再优化最终展示。**

---

## 2. 当前架构核心症结

当前链路大致是：

- 微信抓取 / 官网抓取
- 各自提取文本、图片、OCR/视觉结果
- 合并为标准 JSON
- Word 生成阶段再次做清洗、修复、摘要化、来源猜测

这导致两个后果：

1. **中间层数据不是“可直接消费的标准成品”，而是“待修补原料”**
2. **Word 成为内容质量问题第一次被系统性看见的地方**

因此，下一阶段不是继续在 Word 层堆规则，而是要重构成：

- 上游只负责采集原始信息
- 标准化层统一做分类、结构化、来源识别、置信度评估、异常标记
- 审核层单独输出“人工只需看什么”
- Word 层只负责呈现，不再承担主要内容修复

---

## 3. 问题 → 方案设计总览

下面每一条问题都给出对应方案设计、落地思路和实施重点。

### 问题 1：标准 schema 过薄，缺乏可审核能力

#### 现象
当前 `CreditCardItem` 只有 `bank/title/raw_text/structured` 这类浅层字段。

因此无法回答这些关键问题：
- 这个标题是原文标题还是自动改写的？
- 这个来源是原始来源还是系统猜出来的？
- 这个字段来自正文、OCR 还是图片摘要？
- 这条内容是否高风险、是否需要人工复核？

#### 方案设计
扩展 schema，新增“中间层质量信息”。

建议新增字段：

```json
{
  "issuer_bank": "农业银行",
  "source_name": "农业银行信用卡",
  "source_type": "wechat|website|ocr|unknown",
  "title_source": "raw|generated|rewritten",
  "confidence": {
    "category": 0.92,
    "issuer_bank": 0.78,
    "title": 0.83,
    "structured": 0.74
  },
  "evidence": {
    "issuer_bank": ["银行：中国农业银行", "标题含 农行"],
    "category": ["标题命中 首发", "OCR 命中 卡种"]
  },
  "noise_flags": ["ocr_noise", "unknown_source", "nav_content_pollution"],
  "review_flags": ["needs_source_review", "needs_title_review"],
  "content_blocks": []
}
```

#### 具体思路
1. 保留现有字段兼容旧逻辑
2. 先增量加入新字段，不立即推翻旧数据结构
3. 所有下游只读新字段，旧字段逐步退役
4. `review_flags` 作为人工审核入口字段

#### 预期收益
- 人工不再需要全文判断“这条是否可信”
- 后续可按 flag 定向审核
- Word 可只展示已清洗后的稳定字段

### 问题 2：主流程与标准化脚本并存，形成双轨逻辑

#### 现象
当前 `_agent.py` 内自己做：
- 微信分类
- structured 生成
- 标题改写

同时 `news-analyzer/scripts/convert_to_standard.py` 和 `wechat-article-extractor/scripts/convert_to_standard.py` 也在做标准化。

这意味着同一条内容可能出现两套不同结果。

#### 方案设计
建立**唯一标准化入口**，所有来源都必须先进入统一标准化服务，再进入后续流程。

建议新增模块：
- `common/normalizer.py`
- `common/review.py`

结构建议：

```python
normalize_item(raw_item, source_type) -> StandardItem
classify_item(item) -> category
extract_entities(item) -> issuer/source/evidence
build_structured(item) -> structured fields
score_item(item) -> confidence/review flags
```

#### 具体思路
1. `_agent.py` 不再直接拼 `structured`
2. `convert_to_standard.py` 也不各自维护独立分类规则
3. 所有来源只产出“原始采集结果”
4. 统一交给 normalizer 做：
   - 分类
   - 来源识别
   - 标题清洗
   - structured 生成
   - review flags 生成

#### 预期收益
- 所有入口统一口径
- 修一个问题，全链路生效
- 降低未来维护和排查成本

---

### 问题 3：Word 层承担了过多内容修复职责

#### 现象
当前 Word 生成阶段承担了：
- 来源猜测
- 标题摘要化
- OCR 噪音清除
- 详情压缩
- 公告句子清洗
- 图片广告过滤

这说明 Word 当前不是展示层，而是“最终修复层”。

#### 方案设计
重构责任边界：

- **采集层**：抓原始内容
- **标准化层**：清洗、结构化、分类、打标签
- **审核层**：筛出人工要看的内容
- **展示层（Word）**：只拼版，不修内容

#### 具体思路
1. 将下面逻辑上移到标准化层：
   - `_resolve_highlight_source_name()`
   - `_normalize_detail_text()`
   - 权益变更首句清洗
   - 新卡营销尾巴裁剪
2. `generate_report.py` 未来只消费：
   - `display_title`
   - `display_source`
   - `display_summary`
   - `structured_clean`
3. 把展示字段在标准化阶段直接产出

建议新增字段：
- `display_title`
- `display_source`
- `display_highlight`
- `structured_clean`

#### 预期收益
- Word 层大幅简化
- 同一份标准数据可复用到别的输出方式
- 问题更早暴露，不必等生成文档才发现

---

### 问题 4：分类逻辑脆弱且分散，导致内容错位

#### 现象
当前分类主要靠关键词，且分散在多个脚本。
这会导致：
- 活动误判成公告
- 权益变更误判成活动
- 其他类被误塞进已有类别

一旦分类错，下游标题、亮点、字段模板都会错。

#### 方案设计
建立统一分类器，采用“规则优先 + 打分输出 + 低置信度回审”的方式。

#### 具体思路
分类器分三层：

1. **强规则层**
   - 明确标题模式直接判定
   - 如：`关于...授权书` 高概率权益变更
   - `首发/新卡/发行/上市` 高概率新卡

2. **弱规则打分层**
   - 标题关键词得分
   - 正文关键词得分
   - OCR/图片摘要关键词得分
   - 来源站点加权

3. **低置信度降级层**
   - 分数接近时直接标记 `needs_category_review`
   - 不强行自动判一个“看似正确”的类别

建议输出：

```json
{
  "category": "活动",
  "category_candidates": [
    ["活动", 0.71],
    ["公告", 0.58]
  ],
  "review_flags": ["needs_category_review"]
}
```

#### 预期收益
- 分类稳定性提升
- 错分不再悄悄进入成品 Word
- 人工只看边界样本

---

### 问题 5：微信图片/OCR 结果与正文混成一个字符串，污染全链路

#### 现象
当前 OCR/视觉提取结果会直接进入 `raw_text` 或 `structured[详情]`。
这会导致：
- 图片说明和正文混杂
- 无价值图片内容污染摘要
- Word 亮点、详情、标题都被连带影响

#### 方案设计
建立**块级内容模型**，不要再把不同层次信息拼成一个大字符串。

建议 `content_blocks` 结构：

```json
[
  {"type": "article_text", "text": "正文段落...", "confidence": 0.95},
  {"type": "ocr_fact", "text": "银行：中国农业银行", "confidence": 0.86},
  {"type": "ocr_noise", "text": "图片为山水插画", "confidence": 0.22},
  {"type": "image_cta", "text": "扫码申卡", "confidence": 0.30}
]
```

#### 具体思路
1. 微信上游不再直接拼 `full_text += 图片分析`
2. 图像提取结果先落为独立 block
3. 标准化层只消费：
   - `article_text`
   - `ocr_fact`
4. `ocr_noise` / `image_cta` 只打 flag，不进入展示字段

#### 预期收益
- OCR 噪音不会污染整条内容
- 后续提取可针对 block 精细处理
- 新卡/活动类的结构化质量明显提升

---

### 问题 6：来源模型不清，银行名、公众号名、作者名混用

#### 现象
当前经常出现：
- `未知公众号`
- 来源是 URL
- 来源是标题猜测
- bank 和 author 含义不清

#### 方案设计
将“谁发的”和“谁相关”拆开建模。

建议字段：
- `issuer_bank`：实际关联银行
- `publisher_name`：发布主体，如公众号/官网栏目
- `source_name`：展示用来源名
- `source_type`：wechat / website / bank_site / unknown

#### 具体思路
来源识别优先级：

1. 官网抓取：银行配置直接给 `issuer_bank`
2. 微信号：
   - 先识别公众号名是否包含银行
   - 再从标题/正文/OCR 交叉确认
3. 如果 `publisher_name != issuer_bank`，两者都保留
4. 展示层统一使用 `display_source`

示例：

```json
{
  "issuer_bank": "农业银行",
  "publisher_name": "农行信用卡",
  "source_name": "农业银行",
  "source_type": "wechat"
}
```

#### 预期收益
- 消灭 Word 中的大量“未知公众号”展示问题
- 来源展示更自然
- 人工无需手工替换来源名

---

### 问题 7：官网正文清洗依赖字符串规则，扩展性差

#### 现象
当前网站正文污染主要靠：
- 关键词过滤
- 行过滤
- 页脚/导航排除

这类方法有效，但随着银行站点增加，维护成本会持续上升。

#### 方案设计
从“纯文本清洗”转向“正文块提取 + 站点策略配置”双层机制。

#### 具体思路
1. 抓取层输出：
   - 页面标题
   - 正文候选块
   - 导航候选块
   - 下载区候选块
2. 对每个银行配置：
   - 主正文 selector
   - 噪音 selector
   - 特殊起始/结束 marker
3. 先在 DOM 层过滤，再到文本层补充清洗
4. 将污染检测结果写入 `noise_flags`

建议配置化：

```python
BankConfig(
  detail_content_selector=".notice-content",
  detail_remove_selectors=[".breadcrumb", ".footer", ".hotline", ".download-list"],
  content_start_markers=["尊敬的客户", "关于", "我行"],
)
```

#### 预期收益
- 导航页污染明显减少
- 新银行接入时不必大量堆 if/else
- 维护方式更清晰

---

### 问题 8：缺少“人工审核视图”，导致必须全文复核

#### 现象
现在系统只有“最终 Word”，没有“告诉你哪里需要看”的审核产物。

#### 方案设计
新增审核输出层，单独产出审核清单。

建议输出文件：
- `review_queue.json`
- `review_queue.md`

#### 审核清单内容设计
每条只显示高风险信息：
- 标题是否自动改写
- 来源是否猜测得出
- 分类是否低置信度
- 是否含 OCR 噪音
- 是否缺少时间/适用人群/活动门槛等关键字段
- 原文链接
- 对应证据

示例结构：

```json
{
  "item_id": "4aa2ded907e0",
  "title": "农行Visa全球支付白金卡世界杯版首发",
  "review_flags": ["needs_source_review", "needs_detail_review"],
  "confidence": {"issuer_bank": 0.72, "structured": 0.61},
  "evidence": {"issuer_bank": ["OCR: 银行：中国农业银行"]},
  "url": "..."
}
```

#### 具体思路
1. 审核层不修改内容，只做筛选和汇总
2. Word 生成前先输出审核清单
3. 后续可以支持只生成“高风险条目复核稿”

#### 预期收益
- 人工从“全文通读”转为“按异常抽查”
- 审核效率显著提升
- 便于后续对问题类型做统计和迭代

---

### 问题 9：标题与亮点生成仍偏补丁式，难以稳定产出编辑感

#### 现象
当前亮点和标题优化已经有所改善，但仍然依赖下游按类别做临时裁剪和重写。

#### 方案设计
建立“展示文案层”，把可编辑字段与原始字段分离。

建议字段：
- `raw_title`
- `normalized_title`
- `display_title`
- `highlight_summary`
- `editor_note`

#### 具体思路
1. 原始标题永远保留
2. 标准化层生成：
   - `normalized_title`：去噪但尽量保留原意
   - `display_title`：给 Word 正文用
   - `highlight_summary`：给本期亮点用
3. 不再让 Word 动态从 `raw_text` 临时抽句子
4. 分类模板化生成展示文案：
   - 新卡：银行 + 卡种 + 动作 + 主题
   - 权益变更：动作 + 对象 + 生效信息
   - 活动：主题 + 期次/奖励点

#### 预期收益
- 编辑感更稳定
- 不同输出场景可直接复用
- Word 不再承担文案推导职责

---

### 问题 10：项目层次混杂，运行链路与职责边界不清

#### 现象
当前项目根目录下：
- `_agent.py` 做大量流程编排和业务判断
- 多个脚本既能单跑又部分重复
- data、文档、工具、产物混在一起

这会让后续重构越来越难。

#### 方案设计
按职责重新组织模块：

```text
common/
  schema.py
  normalizer.py
  classifier.py
  entity_resolver.py
  review.py
  display_fields.py

sources/
  wechat/
  website/

pipelines/
  fetch_pipeline.py
  normalize_pipeline.py
  review_pipeline.py
  report_pipeline.py

outputs/
  word/
  review/
```

#### 具体思路
1. `_agent.py` 逐步瘦身，仅保留 orchestration
2. 分类、来源识别、标题生成独立模块化
3. 各源采集与各类输出解耦
4. 旧脚本保留兼容入口，内部改为调用新模块

#### 预期收益
- 更容易测试和替换某一段逻辑
- 降低新功能迭代难度
- 问题归因更清晰

## 4. 下一阶段总体重构方案

### 阶段 A：建立稳定中间层（最高优先级）

目标：让系统先知道“哪些内容可信、哪些要审、哪些是噪音”。

#### 要做的事
1. 扩展 `common/schema.py`
2. 新建统一标准化模块 `common/normalizer.py`
3. 引入 `confidence / evidence / flags`
4. 引入 `display_*` 展示字段
5. 引入 `content_blocks`

#### 完成标志
- 任意来源进入系统后，都有统一标准项
- Word 不再需要猜来源和大幅清洗内容
- 可以输出 review queue

---

### 阶段 B：统一分类、来源识别、结构化提取

目标：让内容质量问题在标准化层解决，而不是在 Word 暴露。

#### 要做的事
1. 抽离分类器模块
2. 抽离来源/银行识别模块
3. 抽离 structured 生成模块
4. 将微信 OCR、官网正文都转换为 block 后再消费
5. 保留 raw 数据，但不直接用于展示

#### 完成标志
- `_agent.py` 不再自行构造 `structured`
- `convert_to_standard.py` 只负责适配原始输入，不再各自维护规则
- 标题/摘要/来源由统一模块产出

---

### 阶段 C：建立审核层与高风险流

目标：把人工工作量从“全量通读”降到“看异常项”。

#### 要做的事
1. 新建 `common/review.py`
2. 根据 flags 和 confidence 输出审核队列
3. 输出 `review_queue.md/json`
4. Word 里可附加“待复核事项”页，或单独产出一份复核报告

#### 完成标志
- 运行完后除了 Word，还有一份审核清单
- 人工只需先看有 flags 的条目

---

### 阶段 D：简化 Word 层为纯展示层

目标：让 Word 生成器只做呈现，不做重型清洗和猜测。

#### 要做的事
1. `generate_report.py` 只读 `display_title/display_source/highlight_summary/structured_clean`
2. 将当前大量清洗函数逐步迁出
3. 保留极少量纯排版逻辑：
   - 图片尺寸控制
   - 页面结构
   - heading 样式

#### 完成标志
- Word 代码显著瘦身
- 文档问题基本能追溯到上游标准化结果，而非 Word 临时逻辑

---

## 5. 具体实施顺序（建议按低风险推进）

### 第 1 批：最小重构，先建立可审核性

1. 扩 schema，新增：
   - `issuer_bank`
   - `source_name`
   - `source_type`
   - `confidence`
   - `evidence`
   - `noise_flags`
   - `review_flags`
2. 在现有链路上补这些字段，不改 Word 入口
3. 输出 `review_queue.json/md`

**目的**：先让系统能告诉你哪里不可信。

---

### 第 2 批：统一标准化口径

1. 新建 `common/normalizer.py`
2. `_agent.py` 改为调用 normalizer
3. 把分类、来源识别、title/display 字段统一迁入
4. 旧 `convert_to_standard.py` 改为薄适配器

**目的**：消除双轨逻辑。

---

### 第 3 批：重构微信/OCR 和官网清洗策略

1. 微信图像结果改为 block 输出
2. 官网正文改为 DOM 层 + 文本层双层过滤
3. structured 从 block 构建
4. 噪音只进 flags，不进展示字段

**目的**：消除 Word 中最常见的内容污染根因。

---

## 7. 对人工审核流程的直接改造建议

重构不是为了“代码更优雅”，而是为了减少你的手工修改时间。

建议新的人工流程变成：

1. 运行抓取和标准化
2. 先看 `review_queue.md`
3. 只处理：
   - 来源不明
   - 分类不稳
   - OCR 噪音高
   - structured 缺关键字段
4. 再生成 Word
5. 最后只做快速视觉检查，而不是全文重校

理想状态下，人工时间分配应从：
- 80% 全文校对
- 20% 局部修饰

逐步变成：
- 20% 风险项审核
- 80% 直接产出可用

## 8. 验收标准

下一阶段重构是否成功，不看“代码写了多少”，而看下面几个指标：

### 数据层指标
- 同一条内容从不同入口进入后，分类结果一致
- `未知公众号` 显著减少
- OCR 噪音不再进入展示字段
- 官网导航污染进入 `noise_flags` 而不是正文

### 审核效率指标
- 可以自动输出 review queue
- 人工无需通读所有条目
- 低置信度内容可单独复核

### 成品质量指标
- Word 本期亮点更像编辑摘要，而不是原始标题清单
- 正文来源、详情、活动内容更稳定
- 手工修改次数和修改幅度明显下降

---

## 9. 最终建议：下一阶段只抓三件最重要的事

如果下一阶段只做最重要的三件事，建议优先级如下：

### 第一优先级：统一标准化入口
原因：不统一口径，任何优化都会重复和打架。

### 第二优先级：引入 review flags / confidence / evidence
原因：这是减少人工审核成本的关键，不做这层，系统永远只能靠人成品验收。

### 第三优先级：内容块化，切断 OCR/导航噪音污染链
原因：这是当前 Word 中大多数“看起来像展示问题”的真正根因。

---

## 10. 结语

这次重构的重点，不应再是继续修 Word 表现层细节，而应转向：

**把“内容是否可靠、是否需要人工看、该怎么展示”在进入 Word 之前就决定好。**

只有这样，这个项目才会从“自动生成一份还需要人工深改的周报”，升级成“自动生成 + 人工只审异常项”的真正省力工具。
